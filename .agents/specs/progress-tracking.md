# Spec: Progress Tracking

> **Source fidelity: 70%** — The concept of a persistent state file driving the loop is core Huntley (IMPLEMENTATION_PLAN.md, fix_plan.md). The Ralph Playbook repo documents this pattern explicitly. The specs post confirms SPECS.md as an overview document with a table linking to all specs — `progress.md` serves this same "overview with links to details" purpose for learner state. The specific table structure, status categories, and cleanup rules are our design.

## Purpose

Progress tracking is the persistent state file that drives the entire learning loop. Every other spec reads from it and writes to it. Without it, every session starts from zero.

## Why This File Exists

In Huntley's Ralph system, `IMPLEMENTATION_PLAN.md` is the file that:
- Persists between loop iterations (survives context window reset)
- Tells the agent what's done and what's remaining
- Drives what the agent works on next
- Is disposable — can be regenerated from scratch
- Must be kept current after every session

Source: ghuntley.com/ralph/ — "Always keep @fix_plan.md up to date with your learnings using a subagent. Especially after wrapping up/finishing your turn."

Source: Ralph Playbook — "IMPLEMENTATION_PLAN.md: Prioritized bullet-point list of tasks derived from gap analysis (specs vs code) - generated by Ralph. Created via PLANNING mode. Updated during BUILDING mode. Can be regenerated."

Additionally, from ghuntley.com/specs/: "Create a SPECS.md in the root of the directory which is an overview document that contains a table that links to all the specs." This overview pattern — a single file with a table linking to details — maps directly to `progress.md`: a single file with a table of all topics linking to detailed findings.

`progress.md` is the learning equivalent of both `IMPLEMENTATION_PLAN.md` (persistent state) and `SPECS.md` (overview with table).

## File Location

```
progress.md    # at project root, single source of truth
```

## Structure

### Section 1: Topic Overview

High-level summary of all topics and current competency.

```markdown
## Topic Overview

| Topic                  | Current Tier | Avg Score | Last Studied  | Status       |
|------------------------|-------------|-----------|---------------|--------------|
| Linux Fundamentals     | Tier 3      | 4.2/5     | 2026-03-07    | Strong       |
| Storage Fundamentals   | Tier 2      | 2.8/5     | 2026-03-08    | Needs Work   |
| Networking Fundamentals| Tier 1      | 1.5/5     | 2026-03-06    | Weak         |
```

Status mapping:
- **Strong** (avg ≥ 4.0) — Approaching mastery. Retention checks only.
- **Decent** (avg 3.0-3.9) — Solid foundation. Push to higher tiers.
- **Needs Work** (avg 2.0-2.9) — Core gaps. Priority for study.
- **Weak** (avg < 2.0) — Significant gaps. Highest priority.
- **Not Started** — Exists in readings but not yet assessed.

### Section 2: Detailed Findings

Per-topic strengths, weaknesses, and specific misconceptions.

```markdown
### Linux Fundamentals

**Strengths:**
- {specific things the learner demonstrated well}

**Weaknesses:**
- {specific gaps identified through grading}

**Misconceptions identified:**
- {specific wrong beliefs that need correcting}

**Recommended focus:**
- {targeted areas for next session}
```

### Section 3: Session History

Chronological log enabling trend analysis.

```markdown
## Session History

| Date       | Topic         | Type        | Score  | Notes                     |
|------------|---------------|-------------|--------|---------------------------|
| 2026-03-07 | Linux         | Interactive | 4.5/5  | Strong on permissions     |
| 2026-03-08 | Storage       | Quiz        | 2.8/5  | LVM needs work            |
```

## Update Rules

**From Huntley (directly sourced):**
- Update after EVERY interaction — "Especially after wrapping up/finishing your turn"
- When it becomes large, clean out completed items — "When @fix_plan.md becomes large periodically clean out the items that are completed"
- It's disposable — if it gets stale, regenerate from scratch
